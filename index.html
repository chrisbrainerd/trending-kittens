  • Why does chrome say you need to access my browser history?

    So it's about to get techie real fast, but the tl;dr up front is that I'm not looking at or storing any history data, but because facebook works in a fancy way I have to watch to see when your history changes to know when to put a gif back on the page.

    The long version for those who care is that Facebook is built on React.js, and more importantly is built as a Single Page App (SPA) and uses its own routing. The old way would be like this: you go to facebook.com and it sends a request to the server which sends you back a full .html file with everything on facebook.com you end up seeing - this triggers the window.onload() signal (and a few others), and so we could just set our gif-placer doodad in there. If you went to your profile, it'd do the same - sending a new request to the server, sending back a .html, and triggering window.onload(). However, the way React (and many other frontend frameworks) work is that when you go from facebook.com to your profile, it sends a request, but then the server only sends back the parts that changed, and there's some javascript magic that updates the DOM. Because we're just hot-swapping DOM components, it never triggers window.onload() again. However, it does trigger onHistoryStateUpdated (which is why when you navigate to your profile, you can hit back and return to facebook.com). In order to use that signal, chrome extensions need to add the 'webNavigation' permission, which is what's giving you that scary message.
